# NVIDIA Omniverse + NIM on AWS — 디지털 트윈 PoC 발표 자료

작성일: 2026-05-30 · 발표자: yeonkp

## 1. 한 줄 요약

> 온프렘에서 막혔던 NVIDIA Omniverse(3D 디지털 트윈) 구동을, AWS GPU 인스턴스 위에
> 코드(CDK)로 재현 가능하게 만들고 실제로 화면이 뜨는 것까지 검증했다.

---

## 2. 왜 했나 — 풀어야 했던 문제

산업 설비를 3D 가상 모델로 만드는 "디지털 트윈"을 우리 환경에서 구동하려 했으나,
온프렘 R&D 서버에서 NVIDIA Omniverse 설치가 중간에 막혔다.

| 온프렘에서 막힌 이유 | 실제 원인 |
|---------------------|-----------|
| NIM(AI 서비스)은 설치됐는데 Kit(렌더링 엔진)이 안 됐다 | GPU 한계가 아니라 환경 제약 |
| 외부 인터넷 차단 | NVIDIA 패키지/이미지 다운로드 불가 |
| 관리자 권한 없음 | 사용자가 직접 설치 못 함 |
| 개인 PC Docker 제한 | 실행 환경 자체가 제약 |

핵심 진단: 실패는 GPU 성능 문제가 아니라 "인터넷 차단 · 권한 · 개인PC" 환경 제약이었다.
→ AWS EC2로 옮기면 이 세 가지가 구조적으로 해소된다 (root 권한 보유, 아웃바운드 허용 가능,
EC2 자체가 Docker 호스트).

---

## 3. 왜 AWS인가 — 디지털 트윈 활용 측면

디지털 트윈 워크로드는 "무거운 GPU가 가끔·불규칙하게, 그러나 강력하게" 필요하다.
이 특성에 AWS가 잘 맞는다.

| 디지털 트윈의 특성 | 온프렘의 한계 | AWS에서의 이점 |
|--------------------|---------------|----------------|
| 최신·고성능 GPU 필요 | 조달에 수개월·고정 투자, 노후화 위험 | RTX PRO 6000(Blackwell)·L40S를 분 단위로 즉시 사용 |
| 사용이 불규칙 (집중 작업 후 유휴) | 안 써도 하드웨어 비용 고정 | 미사용 시 stop → 인스턴스 비용 0, 쓴 만큼만 |
| 워크로드 크기 변동 (씬 규모·동시 사용자) | 최대치에 맞춰 과투자 | 인스턴스 타입만 바꿔 즉시 스케일업/다운, 여러 대 병렬 |
| 패키지·이미지 외부 다운로드 필수 | 폐쇄망·권한 제약으로 설치 자체가 막힘 | 아웃바운드 허용 + root 권한으로 자유 설치 |
| 환경 재현·이관 반복 | 수작업 설치 매번 반복 (삽질) | 코드(CDK)+골든 AMI로 같은 환경 수 분 내 재현 |
| 결과 공유·원격 협업 | 물리 워크스테이션에 묶임 | DCV(원격 데스크톱)+WebRTC(브라우저)로 어디서든 접속 |

활용 측면 핵심 3가지:
- 진입장벽 제거 — GPU 구매·조달 없이 바로 시작. 실패해도 매몰비용이 작다 (실험에 유리).
- 탄력성 — 한 명의 프로토타이핑부터 다중 연구자·대형 씬까지 같은 코드로 확장.
- 재현성·거버넌스 — 검증한 환경을 코드로 고정하고, 보안(IP 제한·암호화·시크릿)을 강제.

## 4. 무엇을 만들었나 — 산출물

AWS CDK(TypeScript) 코드 한 벌. 명령 한 번으로 디지털 트윈 구동 환경을 통째로 배포한다.

배포되는 것:
- GPU 인스턴스 (RTX PRO 6000 또는 L40S) + Ubuntu 22.04
- GPU 드라이버 · Docker · NVIDIA Container Toolkit 자동 설치
- 원격 데스크톱(DCV) + 브라우저 스트리밍(WebRTC) 접속 환경
- NIM(AI 추론 컨테이너) 자동 설치 옵션
- 보안(IP 제한, 시크릿 관리, 암호화) · 모니터링 · 비용 태깅 일체

세 구성요소의 관계:
```
[ USD ]        디지털 트윈의 데이터 포맷 (3D 세계를 기술하는 파일)
   ↑
[ Omniverse ]  USD를 열고·렌더링·시뮬레이션하는 엔진   ← 트윈의 본체 (필수)
   ↑
[ NIM ]        AI로 씬 생성·물리 해석을 돕는 마이크로서비스 (보조)
```

## 5. 핵심 가치 — 이게 왜 의미 있나

### (1) 온프렘에서 막힌 Kit 구동을 AWS에서 검증

온프렘에서 본체(Kit)가 안 떴는데, AWS에서 USD Composer가 RTX 렌더링되는 화면까지 확인.
2026-05-30 g7e.4xlarge(RTX PRO 6000)에서 3D 뷰포트 228 FPS 정상 렌더 검증.

### (2) 검증한 설치 절차를 코드로 고정

한 번 검증한 설치 절차를 CDK 코드 + 골든 AMI로 고정.
다음부터는 부팅 수 분이면 같은 환경이 재현된다 (수작업 설치 반복 제거).

### (3) AI 추론으로 시뮬레이션 수 시간 → 수 초

전통 시뮬레이션(유동·구조·열 등 CAE 해석)은 수 시간~수일. NIM(AI surrogate)은 수 초.
설계 파라미터를 바꿔가며 결과장(압력·응력·온도 등)을 거의 실시간으로 탐색 → 데모 포인트.

```text
사용자가 형상 변경 → NIM AI 추론(수 초) → 3D 시각화 → 즉시 다음 시도
```

### (4) 보안 가드레일 내장

`0.0.0.0/0`(전체 인터넷 개방) 입력 시 합성 단계에서 차단.
운영(사내망 전용)/테스트(특정 IP만) 두 모드를 코드 한 벌로 토글.

## 8. 데모 시나리오 (발표 중 보여줄 것)

1. CDK 명령 한 줄로 환경 배포 (또는 이미 떠 있는 인스턴스)
2. DCV로 접속 → USD Composer에서 3D 산업 설비 씬 열기 (RTX 렌더링)
3. (옵션) 형상 파라미터 변경 → NIM 추론 → 결과장(압력·응력·온도 등) 실시간 시각화
4. 브라우저(WebRTC)로 같은 화면을 팀원에게 공유

## 9. PoC 범위와 한계 — 연구 운영화로 가려면

이 PoC의 목적은 "온프렘에서 막혔던 구동이 우리 환경에서 되는가"의 타당성 검증이다.
그 범위에서는 충분히 동작하지만, 상시 연구 플랫폼으로 쓰려면 추가 투자가 필요하다.

| 용도 | 현재 충분한가 |
|------|--------------|
| 단일 연구자 실험·프로토타이핑 | 충분 |
| "우리 환경에서 되나" 타당성 검증 (= PoC 목적) | 충분 |
| 팀 단위 협업 연구 (같은 씬 동시 편집) | 부족 — Nucleus 협업 서버 필요 |
| 다중 사용자 상시 연구 플랫폼 | 부족 — 멀티유저·스케줄링·EKS 이관 필요 |
| 대상 도메인 AI 예측 연구 | 부족 — 도메인 모델 학습 별도 (Phase 2) |

의도적으로 PoC 범위 밖에 둔 것:
- 다중 사용자 협업 — Nucleus 미포함(로컬 USD 파일 사용). 동시 편집 불가
- 단일 인스턴스 + 수동 start/stop — 다중 사용자 자동 할당·스케줄링 없음
- 데이터 영속성·실험 재현 체계 — 현재 S3 수동 업로드 수준
- AI 모델 도메인 — 검증에 쓴 모델은 자동차 공력 전용. 타 도메인은 모델 학습 필요(환경이 아닌 모델 과제)
- 앱 빌드 일부 수동 — kit-app-template 최초 생성이 대화형(이후 골든 AMI로 단축)

## 11. 핵심 메시지 (마무리 슬라이드)

- 온프렘에서 막혔던 디지털 트윈 구동을 AWS에서 검증했다.
- 실패 원인은 GPU 성능이 아니라 환경 제약이었고, AWS에서 구조적으로 해소된다.
- 검증한 절차를 코드로 고정해, 같은 환경을 수 분 내 재현할 수 있다.
- AI(NIM)로 기존 수 시간 규모의 해석을 수 초로 단축 — 설계 탐색 속도가 핵심 가치.
